### Q1: 为什么RAG系统中需要进行文本切分？切分的核心目标是什么？

**题目类型**：技术原理类

**问题描述**：用户上传的文档可能是几万字的论文或几十页的报告，为什么不能直接对整篇文档进行向量化和检索？切分的核心目标是什么？

**答案要点**：

**为什么需要切分：**

1. **向量模型Token限制**
   - 大多数Embedding模型有输入长度限制（如512/1024/2048 tokens）
   - 超过限制会被截断，导致信息丢失

2. **检索精度问题**
   - 整篇文档主题可能不聚焦
   - 用户查询与文档整体匹配度可能较低

3. **计算成本考虑**
   - 向量化整篇文档的计算成本高
   - 检索时计算量随文档长度指数增长

4. **上下文窗口利用**
   - LLM的上下文窗口有限
   - 需要精炼的片段作为上下文

**切分的核心目标：**

| 目标 | 说明 | 优先级 |
|------|------|--------|
| 语义完整 | 每个Chunk应表达一个完整的语义单元 | P0 |
| 长度合理 | 控制在目标Token范围内 | P0 |
| 上下文连贯 | 相邻Chunk之间保持语义连贯 | P1 |
| 召回友好 | 便于检索时匹配用户意图 | P1 |

### Q2: 语义完整性在切分中为什么如此重要？有哪些场景会破坏语义完整性？

**题目类型**：技术原理类

**问题描述**：语义完整性是切分的核心原则。什么是语义完整性？为什么保证语义完整这么重要？哪些情况会破坏语义完整性？

**答案要点**：

**语义完整性的定义：**
- 每个Chunk应包含一个独立、完整的语义单元
- 语义单元内句子之间有紧密的逻辑关系
- Chunk可以独立被理解，不需要依赖其他上下文

**为什么重要：**

1. **检索效果**
   - 不完整的Chunk被检索到后，无法提供有效信息
   - 上下文缺失导致答案不准确

2. **用户体验**
   - 用户期望得到完整、有意义的答案
   - 片段化的回答降低使用价值

3. **系统可靠性**
   - 答案质量稳定可预期
   - 减少"无法回答"的情况

**破坏语义完整性的场景：**

| 场景 | 问题 | 示例 |
|------|------|------|
| 标题-正文分离 | 标题丢失导致主题不明 | "## 优点\n文本..." → 只保留正文 |
| 句子中间截断 | 语义不完整 | "RAG系统可以..." → 截断为"RAG系统" |
| 表格跨Chunk | 表头和内容分离 | 只保留表格内容，丢失表头 |
| 列表项拆分 | 列表不完整 | "1. 优点一 2. 优..." → 中间截断 |

### Q3: 如何衡量切分质量？有哪些量化指标？

**题目类型**：技术原理类

**问题描述**：切分效果很难直观评估。如何建立一套量化指标来衡量切分质量？

**答案要点**：

**核心评估指标：**

| 指标 | 计算方式 | 理想值 |
|------|----------|--------|
| Chunk完整性 | 句子边界完整性比例 | >95% |
| 语义连贯性 | 句间语义相似度均值 | >0.7 |
| 平均Chunk长度 | 平均Token数 | 接近target_tokens |
| 长度分布方差 | 长度分布的标准差 | <阈值 |
| 信息密度 | 有效信息Token/总Token | >0.8 |

**语义连贯性计算：**

```python
def calculate_semantic_coherence(chunk_text: str, model) -> float:
    """
    计算Chunk内句子间的语义连贯性
    """
    sentences = split_sentences(chunk_text)
    if len(sentences) < 2:
        return 1.0
    
    embeddings = model.encode(sentences)
    similarities = []
    for i in range(len(embeddings) - 1):
        sim = cosine_similarity(embeddings[i], embeddings[i+1])
        similarities.append(sim)
    
    return sum(similarities) / len(similarities)
```

**实际评估方法：**

1. **自动评估**
   - 长度分布统计
   - 句子边界检测
   - 语义连贯性计算

2. **下游任务评估**
   - 检索召回率测试
   - 问答准确率测试
   - 用户满意度调查

3. **人工抽样评估**
   - 随机抽样人工审核
   - 标注语义完整性问题

### Q4: 什么是Overlap机制？为什么需要Overlap？它的优缺点是什么？

**题目类型**：技术原理类

**问题描述**：在切分时，相邻Chunk之间通常会有一定比例的内容重叠，这就是Overlap。为什么需要Overlap？它有什么代价？

**答案要点**：

**Overlap机制的作用：**

1. **解决边界信息丢失**
   - 用户问题可能涉及Chunk边界
   - 关键信息被截断在两个Chunk中

2. **保持上下文连贯**
   - 相邻Chunk之间有上下文重叠
   - 便于理解完整语义

3. **提高召回质量**
   - 检索命中的Chunk包含更多上下文
   - 减少答案片段化

**Overlap参数建议：**

| 参数 | 建议值 | 说明 |
|------|--------|------|
| overlap_tokens | 80-120 | Token重叠量 |
| overlap_ratio | 15%-20% | 重叠比例 |
| overlap_type | 尾部重叠 | 主要在Chunk尾部添加 |

**Overlap的代价：**

| 代价 | 影响 |
|------|------|
| 存储增加 | 重叠内容重复存储，增加存储成本 |
| 计算增加 | 向量化时重复计算重叠部分 |
| 检索冗余 | 可能检索到语义高度相似的多个Chunk |

**何时使用/不使用Overlap：**

| 场景 | 建议 |
|------|------|
| 叙事性文档（小说、报告） | 使用Overlap |
| 结构化文档（表格、列表） | 谨慎使用或不使用 |
| 代码文件 | 通常不使用 |
| 问答类文档 | 根据Chunk大小灵活处理 |

### Q5: 固定长度切分和语义切分各有什么优缺点？

**题目类型**：技术对比类

**问题描述**：切分策略可以分为固定长度切分和语义切分。这两种方式各有什么特点？适用场景是什么？

**答案要点**：

**固定长度切分：**

**优点：**
- 实现简单，计算速度快
- 输出长度可预测，便于后续处理
- 适合大规模批处理

**缺点：**
- 可能切断句子和段落
- 语义完整性和连贯性较差
- 检索效果不稳定

**语义切分：**

**优点：**
- 保证语义完整
- 检索效果好
- 用户体验更佳

**缺点：**
- 实现复杂
- 计算成本高
- 输出长度不确定

**对比总结：**

| 维度 | 固定长度 | 语义切分 |
|------|----------|----------|
| 实现复杂度 | 低 | 高 |
| 处理速度 | 快 | 慢 |
| 语义完整性 | 差 | 好 |
| 检索效果 | 一般 | 好 |
| 适用场景 | 简单文本、代码 | 复杂文档 |

**实际方案：**
- 语义切分为主要策略
- 固定长度作为兜底（语义切分失败时）
- 参数化配置，根据文档类型选择

### Q6: 如何判断一个句子边界是否完整？

**题目类型**：技术原理类

**问题描述**：在切分时，经常需要判断是否应该在句子边界处截断。如何准确判断句子边界是否完整？

**答案要点**：

**句子完整性的判断标准：**

1. **标点符号**
   - 句号(。.)、问号(？?)、感叹号(！!)
   - 中文用中文标点，英文用英文标点

2. **语义完整性**
   - 句子成分是否完整（主谓宾）
   - 是否有未闭合的括号、引号
   - 是否有未完成的条件从句

3. **上下文依赖性**
   - 是否依赖前文才能理解
   - 是否引用了后文的内容

**实现方法：**

```python
def is_sentence_complete(sentence: str) -> bool:
    """
    判断句子是否完整
    """
    # 1. 检查结尾标点
    if not sentence.rstrip().endswith(('。', '。', '！', '？', '.', '?', '!')):
        return False
    
    # 2. 检查未闭合的括号
    if sentence.count('(') != sentence.count(')'):
        return False
    
    # 3. 检查未闭合的引号
    if sentence.count('"') % 2 != 0:
        return False
    
    # 4. 检查依赖性词汇
    dependency_words = ['因为', '虽然', '如果', '因为', 'when', 'if', 'although']
    if any(sentence.endswith(word) for word in dependency_words):
        return False
    
    return True
```

**边界case处理：**

| 情况 | 处理方式 |
|------|----------|
| 句子太长超过max_tokens | 强制在子句边界切分 |
| 句子太短（如只有一个词） | 合并到前一个Chunk |
| 列表项 | 保持列表项完整性 |
| 表格单元格 | 不拆分单元格 |

### Q7: 长表格在切分时面临哪些挑战？如何解决？

**题目类型**：场景解决类

**问题描述**：企业文档中的表格可能很长（几十行甚至上百行）。长表格的切分需要特别注意哪些问题？

**答案要点**：

**长表格切分的挑战：**

1. **结构完整性**
   - 表头与内容分离
   - 合并单元格被拆分

2. **上下文丢失**
   - 表格目的和背景信息丢失
   - 跨Chunk的关联性丢失

3. **语义不完整**
   - 独立的表格行难以理解
   - 数据缺乏说明

**解决方案：**

1. **表头重复**
   - 每个Chunk都包含表头
   - 保持列的含义清晰

2. **表格摘要**
   - 每个Chunk包含`table_summary`字段
   - 描述表格整体内容

3. **按行块切分**
   - 保持行数在合理范围（如10-20行）
   - 每个块是独立的"子表格"

**实现示例：**

```python
def split_table(table, target_rows=15):
    """
    按行块切分长表格
    """
    header = table.rows[0]  # 表头
    data_rows = table.rows[1:]
    
    chunks = []
    for i in range(0, len(data_rows), target_rows):
        chunk_rows = data_rows[i:i+target_rows]
        summary = generate_table_summary(header, chunk_rows)
        
        chunks.append({
            "header": header,
            "rows": chunk_rows,
            "summary": summary,
            "row_range": f"{i+1}-{i+len(chunk_rows)}"
        })
    
    return chunks
```

**跨页表格处理：**
- 检测表格是否跨页
- 合并多页表格后再切分
- 生成跨页标记便于追踪

### Q8: 如何处理代码块的切分问题？

**题目类型**：场景解决类

**问题描述**：技术文档中的代码块如果被错误切分，会导致代码不可用。代码块切分有哪些特殊要求？

**答案要点**：

**代码块切分的特殊要求：**

1. **保持语法完整**
   - 函数定义不能被截断
   - 缩进结构必须完整
   - 括号、引号必须配对

2. **独立可运行**
   - 每个Chunk应该是可独立理解的代码单元
   - 保持import语句完整

3. **最小语义单元**
   - 函数级别或类级别切分
   - 避免切断逻辑流程

**切分策略：**

```python
def split_code_block(code_text: str, language: str) -> list[str]:
    """
    代码块切分
    """
    # 1. 识别函数/类边界
    if language == "python":
        pattern = r'^(def |class |async def )'
    elif language == "java":
        pattern = r'^(public |private |protected |class |interface )'
    else:
        pattern = r'^function |^class |^interface '
    
    # 2. 按函数/类分割
    functions = split_by_pattern(code_text, pattern)
    
    # 3. 检查每个单元的长度
    chunks = []
    for func in functions:
        if count_tokens(func) <= MAX_TOKENS:
            chunks.append(func)
        else:
            # 长函数内部按行数切分
            chunks.extend(split_long_function(func))
    
    return chunks
```

**配置建议：**

| 语言 | 切分边界 | max_tokens |
|------|----------|------------|
| Python | 函数定义、类定义 | 600 |
| JavaScript | 函数声明、类声明 | 600 |
| Java | 方法声明、类声明 | 600 |
| SQL | CREATE TABLE、SELECT | 800 |

### Q9: 如何根据不同文档类型选择合适的切分策略？

**题目类型**：技术选型类

**问题描述**：不同类型的文档（论文、报告、网页、代码）结构差异很大。应该如何根据文档类型选择切分策略？

**答案要点**：

**文档类型与切分策略映射：**

| 文档类型 | 结构特点 | 推荐切分策略 | 参数调整 |
|----------|----------|--------------|----------|
| 论文/报告 | 标题层级清晰 | 按标题层级切分 | target_tokens: 600 |
| 小说/书籍 | 章节连贯 | 按章节+段落切分 | overlap_tokens: 100 |
| 网页 | 结构混杂 | HTML标签+段落切分 | min_tokens: 50 |
| 知识库 | QA对形式 | 保持QA对完整 | 不拆分QA对 |
| 技术文档 | 代码+说明混合 | 代码块独立切分 | 代码块特殊处理 |
| 表格文档 | 数据密集 | 按行块切分 | 保持表头重复 |

**策略选择流程：**

```
分析文档结构
    ↓
┌─────────────────────────────────┐
│ 是否有明确标题层级？              │
│ 有 → 按标题层级切分               │
│ 无 → 进入下一步                    │
└─────────────────────────────────┘
    ↓
┌─────────────────────────────────┐
│ 是否为表格密集型？                │
│ 是 → 按行块切分，保持表头          │
│ 否 → 进入下一步                    │
└─────────────────────────────────┘
    ↓
┌─────────────────────────────────┐
│ 是否包含代码块？                  │
│ 是 → 代码块独立，文本语义切分      │
│ 否 → 语义边界切分                 │
└─────────────────────────────────┘
```

**自适应切分：**
- 先分析文档结构
- 自动选择最优切分策略
- 可配置默认策略和兜底策略

### Q10: 切分过程中如何处理小标题（子标题）？

**题目类型**：场景解决类

**问题描述**：小标题在文档中起到承上启下的作用。切分时应该如何处理小标题与正文的关系？

**答案要点**：

**小标题的作用：**
- 标记主题转换
- 概括后续内容
- 帮助理解文档结构

**处理策略：**

1. **标题+正文合并**
   - 每个Chunk包含其归属的标题
   - Chunk开头包含title_path

2. **层级标题保留**
   - 一级标题、二级标题、三级标题
   - 根据层级决定是否单独成Chunk

3. **标题摘要化**
   - 将标题转化为问题的形式
   - 帮助检索匹配用户问题

**实现示例：**

```python
def split_with_headings(document: Document) -> list[Chunk]:
    """
    保留标题结构的切分
    """
    chunks = []
    current_section = ""
    
    for element in document.elements:
        if element.type == "title":
            level = element.level  # 1-6
            
            # 一级标题：可以作为新Chunk的起点
            if level == 1:
                if current_section:
                    chunks.append(create_chunk(current_section))
                current_section = element.text
            
            # 二级及以下标题：添加到当前section
            else:
                current_section += "\n" + element.text
        
        elif element.type == "paragraph":
            current_section += "\n" + element.text
    
    # 处理最后一个section
    if current_section:
        chunks.append(create_chunk(current_section))
    
    return chunks
```

**检索优化：**
- 标题可以作为独立的检索单元
- 用户可能直接搜索标题内容
- 标题帮助理解Chunk的上下文

### Q11: target_tokens、max_tokens、min_tokens这三个参数如何配合使用？

**题目类型**：技术原理类

**问题描述**：切分时通常需要配置target_tokens、max_tokens、min_tokens三个参数。它们之间的关系是什么？如何设置合理的值？

**答案要点**：

**参数定义与关系：**

| 参数 | 定义 | 作用 |
|------|------|------|
| target_tokens | 目标Token数 | 理想的Chunk大小 |
| max_tokens | 最大Token数 | 超过必须强制拆分 |
| min_tokens | 最小Token数 | 过短需要合并 |

**参数关系图：**

```
min_tokens ←──── target_tokens ────→ max_tokens
    ↓              ↓                    ↓
  合并过短      理想大小              强制拆分
  的Chunk       优先达到              超过必须
```

**推荐配置：**

| Embedding模型 | target | max | min | 说明 |
|---------------|--------|-----|-----|------|
| OpenAI text-embedding-3-small | 512 | 800 | 100 | 标准配置 |
| Qwen3-Embedding | 600 | 900 | 120 | 文档推荐 |
| BGE | 512 | 768 | 80 | 中文优化 |

**调整原则：**

1. **检索精度优先**
   - 较小的target提高检索精度
   - 较大的target保留更多上下文

2. **问答效果平衡**
   - Chunk太小：上下文不足
   - Chunk太大：引入噪声

3. **文档类型适配**
   - 长文档：可增大target
   - QA对：target可设较小

### Q12: 如何通过验证集调优切分参数？

**题目类型**：技术原理类

**问题描述**：切分参数的理论值不一定最优。如何通过验证集实验找到最佳的参数组合？

**答案要点**：

**调优流程：**

```
1. 数据准备
   - 收集典型文档样本
   - 准备问答测试集（ground truth）

2. 参数网格设计
   target_tokens: [400, 500, 600, 700, 800]
   overlap_tokens: [0, 50, 80, 100, 120]
   semantic_threshold: [0.5, 0.6, 0.7, 0.8]

3. 批量实验
   for params in parameter_grid:
       chunks = split_documents(params)
       metrics = evaluate(chunks, qa_testset)
       record(params, metrics)

4. 结果分析
   - 找出各指标的Pareto前沿
   - 选择综合最优的参数组合
```

**评估指标选择：**

| 指标 | 衡量内容 | 权重 |
|------|----------|------|
| Recall@K | 正确答案在TopK召回中的比例 | 30% |

### Q13: 什么是语义断点检测？如何实现？

**语义断点的定义：**
- 句子之间语义发生明显转变的位置
- 主题切换的边界

**检测方法：**

```python
def detect_semantic_breaks(sentences: list[str], model) -> list[int]:
    """
    基于句子相似度检测语义断点
    """
    embeddings = model.encode(sentences)
    breaks = []
    
    for i in range(1, len(embeddings)):
        similarity = cosine_similarity(embeddings[i-1], embeddings[i])
        
        # 相似度骤降表示语义断点
        if similarity < SEMANTIC_THRESHOLD:
            breaks.append(i)
    
    return breaks
```

3. **基于模型的方法**

### Q14: 如何处理过短Chunk的合并问题？

**题目类型**：场景解决类

**问题描述**：切分后会产生一些过短的Chunk（如只有一两句话）。这些Chunk应该如何处理？

**答案要点**：

**过短Chunk的危害：**
- 向量稀疏，检索效果差
- 上下文不足，难以回答问题
- 增加数据库存储开销

**合并策略：**

1. **向后合并（优先）**
```python
def merge_short_chunks(chunks: list[Chunk], min_tokens: int) -> list[Chunk]:
    """
    将过短Chunk合并到下一个Chunk
    """
    merged = []
    current = None
    
    for chunk in chunks:
        if current is None:
            current = chunk
        elif count_tokens(current) >= min_tokens:
            merged.append(current)
            current = chunk
        else:
            # 合并到当前Chunk
            current.content += "\n" + chunk.content
            current.metadata.update(chunk.metadata)
    
    if current is not None:
        merged.append(current)
    
    return merged
```

2. **参数约束**
   - 设置min_tokens阈值
   - 低于阈值的Chunk向前或向后合并

3. **语义完整性优先**
   - 不能为了满足长度破坏语义
   - 必要时允许过短Chunk存在

**特殊情况处理：**

| 情况 | 处理方式 |
|------|----------|
| 文档本身很短 | 保留整个文档 |
| 表格/图片 | 独立保留，不强制合并 |
| 列表项 | 保持列表完整，可接受较短 |

### Q15: 如何处理超长Chunk的强制拆分？

**题目类型**：场景解决类

**问题描述**：某些内容（如长段落、长表格）会超过max_tokens限制。需要强制拆分时，应该如何处理？

**答案要点**：

**强制拆分的原则：**

1. **最小伤害原则**
   - 选择伤害最小的位置拆分
   - 优先在段落间、句子间拆分

2. **保留上下文**
   - 为拆分后的Chunk添加摘要
   - 标记Chunk之间的关系

3. **递归拆分**
   - 反复拆分直到满足长度要求
   - 设置最大递归深度防止死循环

**实现策略：**

```python
def force_split(chunk: Chunk, max_tokens: int) -> list[Chunk]:
    """
    强制拆分超长Chunk
    """
    if count_tokens(chunk) <= max_tokens:
        return [chunk]
    
    sentences = split_sentences(chunk.content)
    
    # 按句子递归拆分
    mid = len(sentences) // 2
    left = "\n".join(sentences[:mid])
    right = "\n".join(sentences[mid:])
    
    chunks = []
    if count_tokens(left) > 0:
        chunks.extend(force_split(create_chunk(left), max_tokens))
    if count_tokens(right) > 0:
        chunks.extend(force_split(create_chunk(right), max_tokens))
    
    # 为子Chunk添加关联信息
    for i, c in enumerate(chunks):
        c.metadata["part_index"] = i
        c.metadata["total_parts"] = len(chunks)
        c.metadata["summary"] = generate_summary(chunk.content)
    
    return chunks
```

**特殊内容处理：**

| 内容类型 | 拆分策略 |
|----------|----------|
| 长段落 | 按句子边界拆分 |
| 长表格 | 按行块拆分，保留表头 |
| 长列表 | 按列表项拆分 |
| 长代码 | 按函数边界拆分 |

```python
def calculate_overlap(sentences: list[str]) -> int:
    """
    根据句子平均长度计算overlap
    """
    avg_sentence_len = sum(len(s) for s in sentences) / len(sentences)
    
    # 保留1-2个完整句子作为overlap
    overlap_sentences = 2
    return int(avg_sentence_len * overlap_sentences)
```

1. **分词差异**
   - 中文需要分词
   - 英文按空格分词
   - 混合文本需要混合处理

2. **语义连贯性**
   - 中英文句子可能表达不同意思
   - 截断位置需要考虑语言边界

3. **专业术语**
   - 技术术语保持完整
   - 变量名、方法名不能拆分

**处理策略：**

1. **语言感知切分**
```python
def split_mixed_content(text: str) -> list[str]:
    """
    中英混合内容的智能切分
    """
    # 检测语言边界
    segments = []
    current_segment = []
    
    for sentence in split_sentences(text):
        if is_chinese(sentence) and is_english(current_segment):
            # 语言切换，开启新段落
            if current_segment:
                segments.append(join_segment(current_segment))
                current_segment = []
        current_segment.append(sentence)
    
    if current_segment:
        segments.append(join_segment(current_segment))
    
    return segments
```

2. **保留英文完整性**
   - 英文单词不拆分
   - 代码片段整体保留
   - 术语保持原样

3. **Embedding适配**
   - 选择支持中英混合的Embedding模型
   - 如Qwen3-Embedding、m3e等

**边界检测规则：**
- 中文句号(。)后可能切分
- 英文句号(.)后可能切分
- 中文引号和英文引号混合时谨慎处理

### Q25: 如何实现自适应的动态切分策略？

**题目类型**：架构设计类

**问题描述**：不同文档、不同章节的内容密度不同。能否实现根据内容密度自动调整切分策略？

**答案要点**：

**自适应切分的设计：**

1. **内容密度分析**
```python
def analyze_content_density(text: str) -> float:
    """
    分析内容密度（有效信息/总文本）
    """
    # 去除模板文本、重复内容
    cleaned = remove_template(text)
    
    # 计算信息熵
    entropy = calculate_entropy(cleaned)
    
    # 计算内容密度
    density = entropy / count_tokens(text)
    
    return density
```

2. **动态参数调整**
```python
def adaptive_chunking(document: Document) -> list[Chunk]:
    """
    自适应切分：根据内容动态调整参数
    """
    chunks = []
    
    for section in document.sections:
        density = analyze_content_density(section.content)
        
        if density > HIGH_DENSITY_THRESHOLD:
            # 高密度内容：切分细一些
            params = SMALL_CHUNK_PARAMS
        elif density > LOW_DENSITY_THRESHOLD:
            # 中等密度：标准切分
            params = NORMAL_CHUNK_PARAMS
        else:
            # 低密度（模板、重复）：粗粒度切分
            params = LARGE_CHUNK_PARAMS
        
        section_chunks = split_with_params(section, params)
        chunks.extend(section_chunks)
    
    return chunks
```

3. **反馈循环**
   - 根据检索效果调整参数
   - 高召回失败率 → 减小Chunk
   - 高噪声率 → 增大Chunk

**参数映射表：**

| 内容类型 | 内容密度 | target_tokens | 策略 |
|----------|----------|---------------|------|
| 干货内容 | 高 | 400 | 细粒度 |
| 标准内容 | 中 | 600 | 标准 |
| 模板内容 | 低 | 800-1000 | 粗粒度 |

### Q26: 如何处理版本更新后的增量切分？

**题目类型**：场景解决类

**问题描述**：当文档更新新版本时，如何只重新切分变化的部分，而不是全量重新切分？

**答案要点**：

**增量切分的思路：**

1. **变更检测**
   - 对比新旧版本的差异
   - 识别新增、修改、删除的区域

2. **影响分析**
   - 变化区域是否影响Chunk边界
   - 是否需要重新生成相邻Chunk

3. **增量执行**
   - 只处理受影响的Chunk
   - 复用未受影响的Chunk

**实现方案：**

```python
def incremental_chunking(old_chunks: list[Chunk], 
                         old_version: Document,
                         new_version: Document) -> list[Chunk]:
    """
    增量切分：只处理版本变化的部分
    """
    # 1. 计算差异
    diff = compute_diff(old_version, new_version)
    
    # 2. 找出受影响的区域
    affected_ranges = get_affected_ranges(diff)
    
    # 3. 确定需要重新切分的Chunk
    chunks_to_reprocess = set()
    for range_start, range_end in affected_ranges:
        for chunk in old_chunks:
            if chunk.overlaps(range_start, range_end):
                chunks_to_reprocess.add(chunk.id)
    
    # 4. 保留未受影响的Chunk
    preserved_chunks = [c for c in old_chunks 
                        if c.id not in chunks_to_reprocess]
    
    # 5. 重新切分受影响区域
    new_chunks = []
    for range_start, range_end in affected_ranges:
        new_content = new_version.get_content(range_start, range_end)
        new_chunks.extend(split_content(new_content))
    
    # 6. 合并结果
    return preserved_chunks + new_chunks
```

**边界处理：**
- 变化区域的边界Chunk需要重新合并
- Overlap部分需要同步更新
- 版本号管理确保可追溯

**策略注册机制：**

```python
class ChunkingStrategy(ABC):
    @abstractmethod
    def split(self, document: Document) -> list[Chunk]:
        pass

class StrategyRegistry:
    _strategies = {}
    
    @classmethod
    def register(cls, name: str, strategy: Type[ChunkingStrategy]):
        cls._strategies[name] = strategy
    
    @classmethod
    def get(cls, name: str) -> ChunkingStrategy:
        return cls._strategies[name]

# 注册策略
StrategyRegistry.register("semantic", SemanticChunkingStrategy)
StrategyRegistry.register("fixed", FixedChunkingStrategy)
StrategyRegistry.register("section", SectionChunkingStrategy)
```

### Q28: 切分服务如何与清洗服务、解析服务协同工作？

**题目类型**：架构设计类

**问题描述**：切分是解析-清洗-切分pipeline的一环。如何设计切分服务与上下游的协同机制？

**答案要点****

**Pipeline协同设计：**

```
解析服务 → 清洗服务 → 切分服务 → Embedding服务
    ↓           ↓           ↓           ↓
DocumentElement CleanDoc  Chunk[]    Vector[]
```

**数据流转：**

```python
# 解析服务输出
document_elements = parse_service.parse(file)

# 清洗服务处理
cleaned_elements = clean_service.clean(document_elements)

# 切分服务处理
chunks = chunk_service.split(cleaned_elements)

# 输出给Embedding服务
for chunk in chunks:
    vector = embedding_service.encode(chunk.content)
    vector_db.insert(chunk.id, vector)
```

**状态管理：**

| 阶段 | 状态 | 说明 |
|------|------|------|
| 解析中 | PARSING | 文档正在解析 |
| 待清洗 | PARSED | 解析完成，等待清洗 |
| 清洗中 | CLEANING | 正在清洗 |
| 待切分 | CLEANED | 清洗完成，等待切分 |
| 切分中 | CHUNKING | 正在切分 |
| 完成 | READY | 所有步骤完成 |

**错误处理协同：**

```python
def pipeline_execute(document: Document):
    try:
        elements = parse_service.parse(document)
    except ParseError as e:
        document.mark_error("PARSE_ERROR", str(e))
        raise
    
    try:
        cleaned = clean_service.clean(elements)
    except CleanError as e:
        document.mark_error("CLEAN_ERROR", str(e))
        raise
    
    try:
        chunks = chunk_service.split(cleaned)
    except ChunkError as e:
        document.mark_error("CHUNK_ERROR", str(e))
        raise
    
    return chunks
```

### Q30: 切分策略如何适配不同的Embedding模型？

**题目类型**：技术选型类

**问题描述**：不同的Embedding模型有不同的输入限制和特点。切分策略应该如何适配不同的Embedding模型？

**答案要点：**

**模型特性对比：**

| 模型 | 最大输入 | 特点 | 切分建议 |
|------|----------|------|----------|
| OpenAI text-embedding-3-small | 8191 | 长上下文 | 可用较大Chunk |
| OpenAI text-embedding-3-large | 8191 | 高精度 | 标准Chunk |
| Qwen3-Embedding | 8192 | 中英双语 | 标准Chunk |
| BGE-large | 512 | 中文优化 | 较小Chunk |
| m3e | 512 | 中文+轻量 | 较小Chunk |

**适配策略：**

```python
def get_chunking_params_for_model(model_name: str) -> dict:
    """
    根据Embedding模型获取切分参数
    """
    model_configs = {
        "text-embedding-3-large": {
            "target_tokens": 800,
            "max_tokens": 1200,
            "overlap_tokens": 100
        },
        "text-embedding-3-small": {
            "target_tokens": 600,
            "max_tokens": 900,
            "overlap_tokens": 80
        },
        "bge-large-zh": {
            "target_tokens": 400,
            "max_tokens": 500,
            "overlap_tokens": 50
        }
    }
    
    return model_configs.get(model_name, DEFAULT_PARAMS)
```

**多模型场景：**

1. **单一模型**
   - 根据模型限制设置参数
   - 优先使用长上下文模型

2. **多模型并存**
   - 为不同模型生成不同Chunk
   - 复用解析结果，按需切分

3. **模型切换**
   - 切分参数与模型解耦
   - 切换模型时重新计算Embedding

**实践建议：**
- 优先使用长上下文Embedding模型
- Chunk大小留有安全边界（80%限制）
- 预留模型升级的扩展性
