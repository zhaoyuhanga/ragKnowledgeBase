# Cursor 项目模块化拆分与清理提示词

## 使用场景

用于让 Cursor Agent 分析当前项目，进行模块化拆分，并清理无用文档和代码。

适合场景：

- Python / FastAPI 后端项目重构
- RAG 知识库项目模块化整理
- 删除旧代码、旧文档、旧实现
- 梳理 API、Service、Repository、Model、Schema 分层
- 在不破坏现有功能的前提下做结构优化

## 推荐使用方式

建议在 Cursor 中使用 **Agent 模式**，将下面的提示词完整粘贴进去。

如果项目较大，建议先执行“第一阶段：只读分析”，确认清单后再让 Cursor 执行删除和移动。

---

## Cursor Agent 提示词

```text
你是资深 Python/FastAPI 架构师。请分析当前项目，进行模块化拆分，并清理无用文档和代码。

重要约束：
这是一个高风险重构任务。必须先分析，不允许一上来删除文件。

执行流程必须严格遵守：
1. 项目扫描
2. 依赖关系分析
3. 模块化拆分方案
4. 无用文件/代码候选清单
5. 等我确认
6. 执行重构和删除
7. 运行验证
8. 修复问题
9. 最终验收

第一阶段：只读分析，不改代码
请先完成以下内容：
1. 扫描项目目录结构。
2. 识别后端、前端、脚本、配置、文档、测试、部署文件。
3. 找出核心启动入口。
4. 找出 API 路由、service、repository、model、schema、core、utils 的边界。
5. 找出所有 import 关系和被引用情况。
6. 找出疑似无用文件，但不要删除。
7. 找出疑似重复代码，但不要修改。
8. 找出模块边界混乱的位置。
9. 输出重构建议和风险等级。

模块化目标：
请将项目整理为清晰分层结构，优先遵循现有框架，不做无意义大改。

后端推荐结构：
app/
  main.py
  api/
    v1/
      routers/
      deps.py
  core/
    config.py
    logging.py
    security.py
    exceptions.py
  db/
    session.py
    base.py
    migrations/
  models/
  schemas/
  repositories/
  services/
    rag/
    documents/
    retrieval/
    llm/
    embedding/
    rerank/
    cache/
  workers/
  utils/
  tests/

RAG 模块推荐结构：
app/services/rag/
  pipeline.py
  chunking/
  retrieval/
  generation/
  citations.py
  evaluation.py

拆分原则：
1. API 层只处理请求、响应、依赖注入，不写业务逻辑。
2. Service 层处理业务流程。
3. Repository 层处理数据库访问。
4. Model 层只放 ORM。
5. Schema 层只放 Pydantic 请求/响应模型。
6. Core 层只放配置、安全、日志、异常。
7. Utils 只能放无业务状态的通用工具。
8. 不允许循环依赖。
9. 不允许把所有东西塞进 common.py / helpers.py。
10. 不允许为了拆分而拆分，保持可运行优先。

无用文件/代码判断标准：
只有满足以下至少两项，才列为删除候选：
1. 没有任何 import 或引用。
2. 不在启动、测试、部署、迁移、脚本链路中。
3. 与当前技术方案冲突，例如旧 embedding、本地 SentenceTransformer、旧切分器。
4. 有明确替代实现。
5. 文档内容过期，且与当前配置或架构冲突。
6. 测试已覆盖替代路径。
7. 删除后验证通过。

禁止删除：
1. migration / alembic 历史文件，除非我明确确认。
2. .env.example / README / docker-compose / Dockerfile。
3. 生产部署脚本。
4. 测试数据和 fixtures，除非确认无引用。
5. 用户上传数据。
6. 任何数据库清理脚本。
7. 任何你不确定用途的文件。

输出格式：
第一阶段请只输出以下内容，不要改代码：

A. 项目结构摘要
B. 当前模块边界问题
C. 建议的新模块结构
D. 文件移动计划
   - 原路径 -> 新路径
   - 影响到的 import
   - 风险等级
E. 删除候选清单
   - 文件路径
   - 删除原因
   - 证据
   - 风险等级
F. 重复代码候选清单
G. 必须保留清单
H. 需要我确认的问题

确认机制：
在我回复“确认执行”之前，禁止：
- 删除文件
- 移动文件
- 大规模改 import
- 修改数据库结构
- 修改 Docker / CI / 部署配置

第二阶段：确认后执行
如果我确认执行，请按以下规则操作：
1. 先做模块移动和 import 修复。
2. 再删除明确无用代码。
3. 每一类改动后运行测试。
4. 测试失败必须修复。
5. 最后运行完整验证。

验证要求：
1. 运行单元测试。
2. 运行 import 检查。
3. 启动 FastAPI 应用验证。
4. 如果有前端，运行前端构建。
5. 如果有 docker-compose，检查配置不被破坏。
6. 输出验证命令和结果。

最终验收标准：
1. 项目可以启动。
2. 测试通过。
3. 无明显循环依赖。
4. 旧无用代码已删除。
5. 模块边界清晰。
6. API 行为不变。
7. 最终回复必须列出：
   - 移动了哪些文件
   - 删除了哪些文件
   - 修改了哪些 import
   - 验证命令
   - 验证结果
   - 剩余风险
```

## 允许低风险直接执行版本

如果你希望 Cursor 不等确认就先做低风险整理，可以使用下面这段替换“确认机制”部分。

```text
本次允许你直接执行低风险重构，但高风险删除必须先列清单等待确认。

低风险包括：
- 移动 service 内部文件
- 修复 import
- 删除明显未引用的临时文件
- 删除空目录
- 删除重复且已有替代实现的非入口代码

高风险包括：
- 删除 migration / alembic 历史文件
- 删除 Dockerfile / docker-compose / CI 配置
- 删除 .env.example / README
- 删除部署脚本
- 删除数据库脚本
- 删除测试 fixtures
- 删除用户上传数据
- 修改数据库结构
- 修改鉴权、安全、权限相关逻辑
```

## 更强限制版本

如果你担心 Cursor 误删文件，可以在提示词末尾追加：

```text
额外限制：
1. 任何删除操作都必须先给出证据。
2. 任何不能确定用途的文件必须保留。
3. 不允许使用 rm -rf 或批量删除命令。
4. 每删除一个文件，都必须说明删除原因和验证结果。
5. 删除后必须运行测试或启动验证。
6. 如果验证失败，必须恢复或修复。
```

## 建议执行节奏

推荐分三轮执行：

1. 第一轮：只读分析，生成模块化方案和删除候选清单。
2. 第二轮：确认低风险文件移动和 import 修复。
3. 第三轮：确认删除旧文档、旧代码、旧实现。

不要把“模块化拆分、删除旧代码、升级 RAG 检索链路、数据库迁移”全部放在同一轮执行。这样风险会非常高，也不利于验收。
