## 一、服务拆分概览

### 1.1 拆分策略

基于业务边界和技术特性，将系统拆分为 **9 个核心微服务**：

| 序号 | 服务名称 | 服务ID | 核心职责 | 优先级 |
|------|----------|--------|----------|--------|
| 1 | API 网关 | `api-gateway` | 统一入口、路由、认证、限流 | P0 |
| 2 | 认证服务 | `auth-svc` | 用户认证、授权、SSO | P0 |
| 3 | 用户服务 | `user-svc` | 用户管理、权限、配额 | P0 |
| 4 | 文档服务 | `doc-svc` | 文档上传、解析、存储 | P0 |
| 5 | 知识库服务 | `kb-svc` | 知识库管理、向量索引 | P0 |
| 6 | 检索服务 | `search-svc` | 检索、重排序、查询处理 | P0 |
| 7 | 问答服务 | `qa-svc` | 问答生成、流式输出 | P0 |
| 8 | 分析服务 | `analytics-svc` | 统计、分析、报表 | P1 |
| 9 | 通知服务 | `notify-svc` | 通知、Webhook、邮件 | P2 |

### 1.2 服务规模配置

```yaml
# 最小部署规模（开发/测试环境）
environments:
  development:
    api-gateway: 1
    auth-svc: 1
    user-svc: 1
    doc-svc: 1
    kb-svc: 1
    search-svc: 1
    qa-svc: 1
    analytics-svc: 1
    notify-svc: 1

  production:
    api-gateway: 3      # 冗余部署
    auth-svc: 3
    user-svc: 3
    doc-svc: 3
    search-svc: 3
    qa-svc: 5          # 重点扩展
    analytics-svc: 2
    notify-svc: 2
```

## 二、核心服务详细设计

### 2.1 API Gateway 服务

```yaml
服务ID: api-gateway
服务名称: API 网关服务
技术栈: Kong Enterprise / Traefik
端口: 80/443

职责:
  - 请求路由分发
  - 身份认证校验
  - API 限流熔断
  - 请求/响应转换
  - 链路追踪接入
  - 日志收集

路由规则:
  /api/v1/auth/*        -> auth-svc
  /api/v1/users/*       -> user-svc
  /api/v1/documents/*   -> doc-svc
  /api/v1/knowledge/*   -> kb-svc
  /api/v1/search/*      -> search-svc
  /api/v1/qa/*          -> qa-svc
  /api/v1/analytics/*   -> analytics-svc
  /api/v1/notify/*      -> notify-svc

限流配置:
  default_limit: 100/min
  authenticated_limit: 1000/min
  premium_limit: 10000/min

健康检查:
  endpoint: /health
  interval: 10s
  timeout: 5s
```

```yaml
  # Token 管理

  DELETE /auth/tokens/{token_id}    # 撤销令牌

    id: UUID

    email: string (unique)
    password_hash: string
    phone: string (nullable)
```

## 三、服务间通信协议

### 3.1 HTTP/gRPC 选择策略

| 场景 | 协议 | 理由 |
|------|------|------|
| 外部 API | HTTP REST | 通用性、兼容性 |
| 服务间同步调用 | HTTP/gRPC | 性能、类型安全 |
| 实时流式响应 | WebSocket | SSE 支持 |
| 异步事件 | Kafka | 解耦、可靠 |

### 3.2 服务发现

```yaml
# 使用 Kubernetes Service + DNS 实现服务发现
service_discovery:
  protocol: DNS
  format: "{service-name}.{namespace}.svc.cluster.local"
  
  # 示例
  auth-svc: "auth-svc.rag-qa-system.svc.cluster.local"
  user-svc: "user-svc.rag-qa-system.svc.cluster.local"
  
# 服务注册中心（Consul/Eureka 备选）
service_registry:
  type: Kubernetes
  health_check: HTTP /health
  interval: 10s
```

### 3.3 消息队列事件设计

```yaml
# Kafka Topic 设计
topics:
  # 文档事件
  doc.document.uploaded:
    partitions: 6
    retention: 7d
    consumers: [kb-svc, analytics-svc]
    
  doc.document.processed:
    partitions: 6
    retention: 7d
    consumers: [kb-svc, notify-svc]
    
  doc.document.failed:
    partitions: 6
    retention: 30d
    consumers: [notify-svc]
  
  # 问答事件
  qa.question.asked:
    partitions: 12
    retention: 7d
    consumers: [analytics-svc]
    
  qa.answer.generated:
    partitions: 12
    retention: 7d
    consumers: [analytics-svc]
  
  # 用户事件
  user.created:
    partitions: 3
    retention: 30d
    consumers: [notify-svc]
    
  user.login:
    partitions: 3
    retention: 30d
    consumers: [analytics-svc]

# 事件消息格式
event_schema:
  event_id: UUID
  event_type: string  # "document.uploaded"
  timestamp: ISO8601
  version: string     # "1.0"
  source_service: string
  correlation_id: UUID  # 用于链路追踪
  user_id: UUID (nullable)
  data: object
```

## 四、服务部署配置

### 4.1 Kubernetes Deployment 模板

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: {SERVICE_NAME}
  namespace: rag-qa-system
  labels:
    app: {SERVICE_NAME}
    version: {VERSION}
spec:
  replicas: {REPLICAS}
  selector:
    matchLabels:
      app: {SERVICE_NAME}
  template:
    metadata:
      labels:
        app: {SERVICE_NAME}
        version: {VERSION}
    spec:
      serviceAccountName: {SERVICE_NAME}
      containers:
        - name: {SERVICE_NAME}
          image: {REGISTRY}/{SERVICE_NAME}:{VERSION}
          ports:
            - name: http
              containerPort: {PORT}
              protocol: TCP
          env:
            - name: SERVICE_NAME
              value: "{SERVICE_NAME}"
            - name: DATABASE_URL
              valueFrom:
                secretKeyRef:
                  name: db-credentials
                  key: url
            - name: REDIS_URL
              valueFrom:
                secretKeyRef:
                  name: redis-credentials
                  key: url
          resources:
            requests:
              memory: "{MEMORY_REQUEST}"
              cpu: "{CPU_REQUEST}"
            limits:
              memory: "{MEMORY_LIMIT}"
              cpu: "{CPU_LIMIT}"
          livenessProbe:
            httpGet:
              path: /health
              port: http
```

## 五、验收标准

### 5.1 拆分验收清单

| 序号 | 验收项 | 验收条件 | 验证方法 |
|------|--------|----------|----------|
| 1 | 服务独立部署 | 每个服务可独立部署，不依赖其他服务 | 单独部署各服务到 K8s |
| 2 | 服务健康检查 | 每个服务实现 /health 和 /ready 接口 | 调用健康检查接口 |
| 3 | 服务发现 | 服务间通过 DNS 发现 | 跨服务调用测试 |
| 4 | 故障隔离 | 单服务故障不影响其他服务 | 模拟服务故障测试 |
| 5 | 水平扩展 | 服务可增加副本数实现扩展 | 增加副本数验证 |
| 6 | 服务通信 | 同步/异步通信正常工作 | 发送消息测试 |
| 7 | 链路追踪 | 请求可追踪到各服务 | 调用链路测试 |
| 8 | 统一日志 | 日志格式统一，可聚合查询 | 日志聚合查询验证 |
