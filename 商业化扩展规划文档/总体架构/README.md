### 1.2 微服务拆分原则

| 原则 | 说明 |
|------|------|
| **单一职责** | 每个服务只负责一个业务领域 |
| **松耦合** | 服务间通过 API 通信，不直接依赖数据库 |
| **高内聚** | 相关功能放在同一服务内 |
| **独立部署** | 每个服务可独立版本发布 |
| **容错隔离** | 单个服务故障不影响整体系统 |

## 二、服务拆分方案

### 2.1 服务列表

| 服务名称 | 服务ID | 职责 | 技术栈 | 依赖服务 | 部署规模 |
|---------|--------|------|--------|----------|----------|
| **API Gateway** | `api-gateway` | 统一入口、路由、认证、限流 | Kong/Traefik | 所有后端服务 | 3+ |
| **Auth Service** | `auth-svc` | 用户认证、授权、SSO | FastAPI + JWT | MySQL, Redis | 3+ |
| **User Service** | `user-svc` | 用户管理、权限、配额 | FastAPI | MySQL, Redis, Auth | 3+ |
| **Document Service** | `doc-svc` | 文档上传、解析、存储 | FastAPI | MySQL, MinIO, Kafka | 3+ |
| **Knowledge Service** | `kb-svc` | 知识库管理、向量索引 | FastAPI | MySQL, Milvus, Redis | 3+ |
| **Search Service** | `search-svc` | 检索、重排序、查询处理 | FastAPI | Milvus, ES, Redis | 3+ |
| **QA Service** | `qa-svc` | 问答生成、流式输出 | FastAPI | LLM API, Redis, Kafka | 3+ |
| **Analytics Service** | `analytics-svc` | 统计、分析、报表 | FastAPI | MySQL, Redis | 2+ |
| **Notification Service** | `notify-svc` | 通知、Webhook、邮件 | FastAPI | Redis, Kafka | 2+ |

## 三、数据库设计规范

### 3.1 数据库选型

| 用途 | 数据库 | 选型理由 |
|------|--------|----------|
| **结构化数据** | MySQL 8.0 | 用户、权限、配置等事务性数据 |
| **向量数据** | Milvus 2.4+ | 高维向量相似度检索 |
| **全文搜索** | Elasticsearch 8.x | 关键词检索、日志分析 |
| **缓存** | Redis 7.0 | 热数据、会话、队列 |
| **文档存储** | MinIO/S3 | 文件、二进制数据 |
| **时序日志** | MongoDB | 审计日志、访问日志 |

### 3.2 库表命名规范

```
命名规范：
- 库名：{project}_{module}_{env}
- 表名：{module}_{entity}
- 索引名：idx_{table}_{column}
- 外键：fk_{table}_{ref_table}

示例：
- 库：rag_qa_user_prod
- 表：user_account, user_permission
- 索引：idx_user_account_email
```

### 3.3 数据隔离策略

| 隔离级别 | 说明 | 适用场景 |
|----------|------|----------|
| **租户隔离** | 不同租户使用独立数据库实例 | 金融、医疗等高合规需求 |
| **Schema隔离** | 同一数据库，不同 Schema | 中等隔离需求 |
| **行级隔离** | 通过 tenant_id 字段隔离 | 一般 SaaS 应用 |

## 四、部署架构

### 4.1 Kubernetes 部署方案

```yaml
# Kubernetes 命名空间
namespaces:
  - name: rag-qa-system
    labels:
      env: production
  - name: rag-qa-monitoring
    labels:
      component: monitoring
  - name: rag-qa-infra
    labels:
      component: infrastructure

# 服务部署配置示例 (Deployment)
apiVersion: apps/v1
kind: Deployment
metadata:
  name: qa-service
  namespace: rag-qa-system
spec:
  replicas: 3
  selector:
    matchLabels:
      app: qa-service
  template:
    metadata:
      labels:
        app: qa-service
        version: v1
    spec:
      containers:
        - name: qa-service
          image: rag-qa/qa-service:v1.0.0
          resources:
            requests:
              memory: "512Mi"
              cpu: "250m"
            limits:
              memory: "2Gi"
              cpu: "1000m"
          livenessProbe:
            httpGet:
              path: /health
              port: 8000
          readinessProbe:
            httpGet:
              path: /ready
              port: 8000
```

### 4.3 多集群部署策略

| 策略 | 说明 | 适用场景 |
|------|------|----------|
| **主备模式** | 一主多备，自动切换 | 跨地域容灾 |
| **多活模式** | 多集群同时服务，按路由分发 | 全球部署 |
| **读写分离** | 主库写、从库读 | 读多写少场景 |
